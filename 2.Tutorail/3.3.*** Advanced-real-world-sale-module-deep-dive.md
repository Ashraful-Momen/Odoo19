```
│       ├── 'views/sale_order_line_views.xml',

    └── ir.model.access.csv
```

> **CRITICAL loading order in `__manifest__.py`:**
> ```python
> 'data': [
>     'views/sale_order_views.xml',   # ← Actions & Views loaded FIRST
>     # ... other view files ...
>     'views/sale_menus.xml',         # ← Menus loaded LAST (they `ref` actions above)
> ]
> ```
> Menus reference actions via `action="action_orders"` — so actions MUST be defined before menus.

### Menu Type Rules (from real code)

| Pattern | Example | Behavior |
|:---|:---|:---|
| Menu WITH `action` only | `<menuitem id="menu_sale_order" action="action_orders"/>` | Clickable leaf — opens the action's screen |
| Menu WITH `name` + `action` | `<menuitem id="menu_sale_order" name="Orders" action="action_orders"/>` | Label from `name`, behavior from `action` |
| Menu WITHOUT `action` | `<menuitem id="sale_order_menu" name="Orders">` | Folder/category — holds child menus |
| Menu WITHOUT `name`, WITH `action` | `<menuitem id="res_partner_menu" action="account.res_partner_action_customer"/>` | Label comes from the action's `name` field |
| `groups=` on menu | `groups="sales_team.group_sale_manager"` | Hidden from users without that group |
| `active="False"` on menu | `active="False"` | Menu hidden until activated by another module |

## 3. action_orders — The Sales Order Action (Full Anatomy)

### Source: `sale_order_views.xml` lines 1003-1017

```xml
<record id="action_orders" model="ir.actions.act_window">
    <field name="name">Sales Orders</field>
    <field name="res_model">sale.order</field>
    <field name="view_mode">list,kanban,form,calendar,pivot,graph,activity</field>
    <field name="search_view_id" ref="sale_order_view_search_inherit_sale"/>
    <field name="context">{'search_default_sales' : 1}</field>
    <field name="path">orders</field>
    <field name="help" type="html">
        <p class="o_view_nocontent_smiling_face">
            Create a new quotation, the first step of a new sale!
        </p><p>
            Once the quotation is confirmed, it becomes a sales order.<br/>
            You will be able to create an invoice and collect the payment.
        </p>
    </field>
</record>
```

### Field-by-Field Explanation

| Field | Value | What It Does |
|:---|:---|:---|
| `name` | `Sales Orders` | Displayed as window title + used as menu label if menu has no `name` |
| `res_model` | `sale.order` | The database model to query |
| `view_mode` | `list,kanban,form,calendar,pivot,graph,activity` | Available view switcher buttons. First (`list`) = default |
| `search_view_id` | `ref="sale_order_view_search_inherit_sale"` | Uses the **inherited** search view (sales-specific filters) |
| `context` | `{'search_default_sales': 1}` | Auto-applies the `sales` filter on load → shows only `state='sale'` |
| `path` | `orders` | URL path segment for browser navigation |
| `help` | HTML | Empty-state message when no records exist |

### What `search_default_sales: 1` Does

```
User clicks "Orders" menu
│
▼
action_orders fires
│
├── context = {'search_default_sales': 1}
│       │
│       └── "Auto-apply the filter whose name='sales'"
│                  │
│                  └── <filter name="sales"
│                           domain="[('state','=','sale')]"/>
│                              │
│                              └── Result: ONLY state=sale records shown
│
└── search_view_id = sale_order_view_search_inherit_sale
        │
        └── "Use this search view (which HAS the 'sales' filter defined)"
```

## 4. ir.actions.act_window.view — Explicit View Attachment

### Source: `sale_order_views.xml` lines 1019-1059

This is the **most important advanced concept**. When Odoo auto-discovers views by `model` matching, it uses **priority**. But when you need EXACT control — "for THIS action, use THAT specific view" — you use `ir.actions.act_window.view`.

```xml
<!-- Attach a SPECIFIC list view for action_orders -->
<record id="sale_order_action_view_order_tree" model="ir.actions.act_window.view">
    <field name="sequence" eval="1"/>
    <field name="view_mode">list</field>
    <field name="view_id" ref="sale.sale_order_list_upload"/>
    <field name="act_window_id" ref="action_orders"/>
</record>

<!-- Attach a SPECIFIC kanban view for action_orders -->
<record id="sale_order_action_view_order_kanban" model="ir.actions.act_window.view">
    <field name="sequence" eval="2"/>
    <field name="view_mode">kanban</field>
    <field name="view_id" ref="sale.sale_order_kanban_upload"/>
    <field name="act_window_id" ref="action_orders"/>
</record>

<!-- Attach a SPECIFIC form view for action_orders -->
<record id="sale_order_action_view_order_form" model="ir.actions.act_window.view">
    <field name="sequence" eval="3"/>
    <field name="view_mode">form</field>
    <field name="view_id" ref="sale.view_order_form"/>
    <field name="act_window_id" ref="action_orders"/>
</record>

<!-- calendar, pivot, graph similarly attached... -->
```

### How It Works — Visual

```
action_orders
│
├── view_mode = "list,kanban,form,calendar,pivot,graph,activity"
│
└── act_window.view records attached:
│
    ┌──────────────────────────────────────────────────────────────┐
    │ sequence=1, view_mode=list → view_id=sale_order_list_upload  │
    │ sequence=2, view_mode=kanban → view_id=sale_order_kanban_upload│
    │ sequence=3, view_mode=form → view_id=view_order_form         │
    │ sequence=4, view_mode=calendar → view_id=view_sale_order_calendar│
    │ sequence=5, view_mode=pivot → view_id=view_sale_order_pivot  │
    │ sequence=6, view_mode=graph → view_id=view_sale_order_graph  │
    └──────────────────────────────────────────────────────────────┘

RESULT: When action_orders is opened:
  - List tab   → sale_order_list_upload (NOT auto-discovered list view)
  - Kanban tab → sale_order_kanban_upload (NOT auto-discovered kanban view)
  - Form tab   → view_order_form
  - Calendar   → view_sale_order_calendar
  - Pivot      → view_sale_order_pivot
  - Graph      → view_sale_order_graph
```

### Why Use act_window.view Instead of Auto-Discovery?

| Auto-Discovery | Explicit act_window.view |
|:---|:---|
| Matches by `model` only | Matches by exact `view_id` |
| Priority-based selection | Sequence-based ordering |
| Same view for ALL actions on this model | DIFFERENT views per action |
| Simple cases | When same model needs different views per menu |

## 5. View Inheritance Chain — The FULL Tree

### 5A. List View Inheritance for "Orders"

This is the chain from the action to the actual rendered list:

```
sale_order_tree                          ← BASE (priority=1000)
│   id="sale_order_tree"
│   <list string="Sales Orders" class="o_sale_order" sample="1">
│       fields: name, date_order, partner_id, user_id, amount_total,
│               state, invoice_status, tag_ids, ...
│   </list>
│
└──▶ view_order_tree                    ← INHERITS sale_order_tree (priority=2, mode=primary)
        id="view_order_tree"
        inherit_id="sale_order_tree"
        │
        │   <list position="attributes">
        │       <attribute name="string">Sales Orders</attribute>
        │   </list>
        │
        └──▶ sale_order_list_upload      ← INHERITS view_order_tree (priority=32, mode=primary)
                id="sale_order_list_upload"
                inherit_id="view_order_tree"
                │
                │   <list position="attributes">
                │       <attribute name="js_class">sale_file_upload_list</attribute>
                │   </list>
                │
                └── THIS is what action_orders uses for its list view!
```

### Source XML for Each Level

**Level 1: Base (`sale_order_tree`, priority=1000)**
```xml
<record id="sale_order_tree" model="ir.ui.view">
    <field name="name">sale.order.list</field>
    <field name="model">sale.order</field>
    <field name="priority" eval="1000"/>
    <field name="arch" type="xml">
        <list class="o_sale_order" string="Sales Orders" sample="1"
              decoration-muted="state == 'cancel'">
            <field name="name" string="Number" readonly="1"/>
            <field name="date_order" optional="show"/>
            <field name="partner_id" readonly="1"/>
            <field name="user_id" widget="many2one_avatar_user"/>
            <field name="amount_total" sum="Total Tax Included"
                   widget="monetary" decoration-bf="1"/>
            <field name="state" widget="badge"
                   decoration-success="state == 'sale'"
                   decoration-info="state == 'draft'" optional="hide"/>
            <field name="invoice_status" widget="badge" optional="show"/>
        </list>
    </field>
</record>
```

**Level 2: `view_order_tree` (priority=2, mode=primary)**
```xml
<record id="view_order_tree" model="ir.ui.view">
    <field name="name">sale.order.list (orders)</field>
    <field name="model">sale.order</field>
    <field name="inherit_id" ref="sale_order_tree"/>
    <field name="mode">primary</field>
    <field name="priority" eval="2"/>
    <field name="arch" type="xml">
        <list position="attributes">
            <attribute name="string">Sales Orders</attribute>
        </list>
    </field>
</record>
```

**Level 3: `sale_order_list_upload` (priority=32, mode=primary)**
```xml
<record id="sale_order_list_upload" model="ir.ui.view">
    <field name="name">sale.order.tree.upload (orders)</field>
    <field name="model">sale.order</field>
    <field name="inherit_id" ref="view_order_tree"/>
    <field name="mode">primary</field>
    <field name="priority" eval="32"/>
    <field name="arch" type="xml">
        <list position="attributes">
            <attribute name="js_class">sale_file_upload_list</attribute>
        </list>
    </field>
</record>
```

### What `mode="primary"` Means

`mode="primary"` tells Odoo: "This inherited view is a **primary view candidate** — eligible for auto-discovery matching, not just a modification patch."

| Without `mode="primary"` | With `mode="primary"` |
|:---|:---|
| Just modifies the parent view in-place | Creates its OWN view entry in the view registry |
| Not auto-discoverable by model matching | IS auto-discoverable (by priority) |
| Only applies as a patch to the base view | Acts as a standalone view option |
| Default behavior for inherited views | Must be explicitly set |

### 5B. List View Inheritance for "Quotations"

```
sale_order_tree                          ← BASE (priority=1000)
│   id="sale_order_tree"
│   <list string="Sales Orders">
│       fields: name, date_order, partner_id, amount_total, state...
│   </list>
│
└──▶ view_quotation_tree                ← INHERITS sale_order_tree (priority=4, mode=primary)
        id="view_quotation_tree"
        inherit_id="sale_order_tree"
        │
        │   Changes made:
        │   ① <list string="Quotations">          ← attributes: rename
        │   ② date_order → REPLACED by create_date ← replace field
        │   ③ state → optional="show"              ← always show state
        │   ④ invoice_status → optional="hide"      ← hide invoice status
        │
        └──▶ view_quotation_tree_with_onboarding  ← INHERITS view_quotation_tree
                id="view_quotation_tree_with_onboarding"
                inherit_id="view_quotation_tree"
                mode="primary"
                │
                │   <list position="attributes">
                │       <attribute name="js_class">sale_onboarding_list</attribute>
                │   </list>
                │
                └── THIS is what action_quotations_with_onboarding uses!
```

**Quotation View — Level 2 source:**
```xml
<record id="view_quotation_tree" model="ir.ui.view">
    <field name="name">sale.order.list (quotes)</field>
    <field name="model">sale.order</field>
    <field name="inherit_id" ref="sale_order_tree"/>
    <field name="mode">primary</field>
    <field name="priority" eval="4"/>
    <field name="arch" type="xml">
        <!-- ① Change the list title -->
        <list position="attributes">
            <attribute name="string">Quotations</attribute>
        </list>
        <!-- ② Replace date_order with create_date -->
        <field name="date_order" position="replace">
            <field name="create_date" string="Creation Date" optional="show"/>
        </field>
        <!-- ③ Always show state column -->
        <field name="state" position="attributes">
            <attribute name="optional">show</attribute>
        </field>
        <!-- ④ Hide invoice_status -->
        <field name="invoice_status" position="attributes">
            <attribute name="optional">hide</attribute>
        </field>
    </field>
</record>
```

### `view_id` directly on action vs `act_window.view`

```xml
<!-- Option A: view_id ON the action = sets the DEFAULT view -->
<record id="action_quotations_with_onboarding" model="ir.actions.act_window">
    <field name="view_id" ref="view_quotation_tree_with_onboarding"/>
    <field name="view_mode">list,kanban,form,...</field>
</record>

<!-- Option B: act_window.view = sets SPECIFIC view per view_mode -->
<record id="some_act_view" model="ir.actions.act_window.view">
    <field name="view_mode">list</field>
    <field name="view_id" ref="some_specific_list_view"/>
    <field name="act_window_id" ref="action_orders"/>
</record>
```

| Method | Scope | When to Use |
|:---|:---|:---|
| `view_id` on action | Sets the **default** view for the action (first view_mode type) | When you want to set what opens initially |
| `act_window.view` | Binds a **specific** view to a **specific** view_mode | Full control — exactly which view for each tab |

## 7. Search View Inheritance — Filter Magic

### 7A. Base Search View

**Source: `view_sales_order_filter` (priority=15)**

```xml
<record id="view_sales_order_filter" model="ir.ui.view">
    <field name="name">sale.order.list.select</field>
    <field name="model">sale.order</field>
    <field name="priority" eval="15"/>
    <field name="arch" type="xml">
        <search string="Search Sales Order">
            <!-- Searchable fields -->
            <field name="name" string="Order"
                filter_domain="['|', '|', ('name', 'ilike', self),
                                         ('client_order_ref', 'ilike', self),
                                         ('partner_id', 'child_of', self)]"/>
            <field name="partner_id" operator="child_of"/>
            <field name="user_id"/>
            <field name="team_id" string="Sales Team"/>

            <!-- Base filter -->
            <filter name="my_sale_orders_filter"
                    string="My Orders"
                    domain="[('user_id', '=', uid)]"/>

            <!-- Group By -->
            <group>
                <filter name="salesperson" string="Salesperson"
                        context="{'group_by': 'user_id'}"/>
                <filter name="customer" string="Customer"
                        context="{'group_by': 'partner_id'}"/>
                <filter name="order_month" string="Order Date"
                        context="{'group_by': 'date_order'}"/>
            </group>
        </search>
    </field>
</record>
```

```xml
<record id="sale_order_view_search_inherit_quotation" model="ir.ui.view">
```

### Three Ways to Filter Data (from real code)

| Method | Example | User Can Remove? |
|:---|:---|:---|
| **Domain on action** | `<field name="domain">[('state','=','sale')]</field>` | ❌ No (hard filter) |
| **Context → search_default_XXX** | `<field name="context">{'search_default_sales': 1}</field>` | ✅ Yes (just a pre-applied filter, user can click to remove) |
| **Search view filter** | `<filter name="sales" domain="[('state','=','sale')]"/>` | ✅ Yes (user clicks to toggle) |

### Real Examples from the Code

**Soft filter (user can remove):**
```xml
<!-- action_orders -->
```

## Final Summary: The 4-Layer Architecture

Every Odoo screen is built from exactly **4 layers**:

```
Layer 1: MENU        →  "Which button in the sidebar?"
                        <menuitem action="..."/>

Layer 2: ACTION      →  "Which model? Which views? What filter?"
                        ir.actions.act_window

Layer 3: MODEL       →  "Which database table?"
                        res_model="sale.order"

Layer 4: VIEW        →  "How to display the data?"
                        ir.ui.view (list/form/kanban/search/...)
                            + View Inheritance (inherit_id + mode=primary)
                            + Explicit Attachment (ir.actions.act_window.view)
```

**And the 3 ways to filter:**
1. `domain` on action → Hard filter (user cannot remove)
2. `context` → `search_default_XXX` → Soft filter (pre-applied, user can remove)
3. `search_view_id` → Search view filters → User-controlled filters

**The key insight from the real `sale` module:**
- Sales Orders and Quotations share the **same model** (`sale.order`)
- They share the **same form view** (`view_order_form`)
- They differ in: **list view** (different inherited versions), **search view** (different filters), **context** (different auto-applied filter)
- The entire difference is controlled by **action configuration**, not model changes
